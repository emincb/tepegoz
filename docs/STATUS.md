## What works end-to-end

- Daemon binds user-scoped Unix socket at `$XDG_RUNTIME_DIR/tepegoz-$uid/daemon.sock` (fallback `$TMPDIR` or `/tmp`). Parent dir `0700` when default path (we own it), socket `0600`. Override paths leave parent perms alone.
- Stale-socket eviction on startup. Refuses to start under another live daemon. Graceful SIGINT shutdown with socket cleanup.
- Wire protocol v4: rkyv-archived `Envelope { version, payload }`, 4-byte big-endian length prefix, bytecheck validation on every read. Messages:
  - Commands: `Hello`, `Ping`, `Subscribe(Status | Docker | DockerLogs | DockerStats)`, `Unsubscribe`, `OpenPane`, `AttachPane`, `ClosePane`, `ListPanes`, `SendInput`, `ResizePane`, `DockerAction`
  - Responses: `Welcome`, `Pong`, `PaneOpened`, `PaneList`, `DockerActionResult`, `Error`
  - Events (in `Event(EventFrame)`): `Status`, `PaneSnapshot`, `PaneOutput`, `PaneExit`, `PaneLagged`, `ContainerList`, `DockerUnavailable`, `ContainerLog`, `ContainerStats`, `DockerStreamEnded`
- PTY manager owns panes; per-pane 2 MiB ring buffer (`VecDeque<Bytes>` with total-byte accounting, oldest-chunks-evict on overflow); per-pane reader + waiter threads; `tokio::sync::broadcast` channel for subscribers.
- Docker subscription: per-`Subscribe(Docker)` task walks platform socket candidates (`$DOCKER_HOST` env > Docker Desktop > Colima > Rancher Desktop > native Linux), pings the first reachable engine, and emits a `ContainerList` immediately plus every 2 s. On unreachable engine: emits `DockerUnavailable { reason }` once per availability transition, retries `Engine::connect` every 5 s. Survives `dockerd` restarts; survives the user starting docker after subscribing.
- Docker lifecycle actions: `Payload::DockerAction { request_id, container_id, kind }` runs in a spawned task so a slow daemon doesn't stall the session loop. Always replies with `DockerActionResult { request_id, container_id, kind, outcome }` â€” engine-unavailable and bollard errors both surface as `Failure { reason }` (verbatim from dockerd; daemon doesn't try to classify). `Remove` is force-remove (`docker rm -f` semantics).
- Docker logs streaming: per-`Subscribe(DockerLogs)` task opens a bollard log stream (stdout + stderr) and forwards each chunk as `Event::ContainerLog { stream, data }`. Always emits a terminal `Event::DockerStreamEnded { reason }` â€” engine unreachable, container exit, container removal â€” so a UI knows the stream is done and never spins waiting for chunks that won't come.
- Docker stats: per-`Subscribe(DockerStats)` task streams bollard stats (~1/sec). CPU% is computed from cpu/precpu deltas using the standard docker-stats-CLI formula; `0.0` whenever the calculation can't be performed (first sample, missing precpu on Windows, sys_delta=0). Like logs, terminates with `DockerStreamEnded`.
- Daemon client session: single dedicated writer task drains an mpsc of outgoing envelopes â€” no per-write locks. **Subscription uniformity**: pane and docker subscriptions share the same `HashMap<id, AbortHandle>` shape (`pane_subs` and `docker_subs`). `Unsubscribe { id }` cancels across both maps with a single check. Phase 4 (Ports/Processes), 5 (SSH remote pty), 7 (port scanner) inherit this shape â€” every new subscription kind adds a map alongside, not a new mechanism. (Through Slice C1 `pane_subs` was a `JoinSet<()>` with no per-id key, so `Unsubscribe` of a pane sub silently no-op'd; the C1 TUI's synthetic re-attach was leaking one zombie forwarder per Scopeâ†’Pane mode switch. Fixed in the C2-gate commit `43b28eb`; regression test `pane_unsubscribe.rs` pins the invariant.)
- TUI architecture: pure-state-machine `App` (in `tepegoz-tui/src/app.rs`) handles every external happening as an `AppEvent` (StdinChunk, DaemonEnvelope, Resize, Tick, PendingActionTimeout) and emits `AppAction`s (SendEnvelope, DrawFrame, FocusTile, Detach{User|PaneExited}, ShowToast{kind,message}). The `AppRuntime` (in `tepegoz-tui/src/session.rs`) executes those actions against the daemon socket, stdin/stdout, and ratatui's terminal. State-machine tests live in `app::tests`; headless render tests in `scope::docker::tests`, `scope::placeholder::tests`, `pty_tile::tests`, and `tile::tests`.
- TUI view model (Decision #7): fixed tiled god view, no mode switching. `View { layout: TileLayout, focused: TileId }`. Five tiles on a 120Ă—40+ terminal: PTY (top full-width), Docker (bottom-left), Ports placeholder "Phase 4" (bottom-middle), SSH Fleet placeholder "Phase 5" (bottom-right), Claude Code placeholder "Phase 9" (bottom-full-width strip). Below 80Ă—24 the layout collapses to a single "Terminal too small" fallback tile. All subscriptions (`AttachPane`, `Subscribe(Docker)`, later Ports/Fleet/ClaudeCode) live concurrently for the life of the TUI.
- TUI focus navigation: `Ctrl-b h/j/k/l` + `Ctrl-b` + arrow keys move focus between adjacent tiles (spatial adjacency with left-align tiebreak, so `Ctrl-b j` from full-width PTY lands on Docker rather than Ports). `Ctrl-b d` / `Ctrl-b q` detach from any tile. The focused tile owns the keystroke stream; unfocused tiles continue to update live. Input routing per tile kind: Pty â†’ `SendInput`; Scope(Docker) â†’ scope key parser (j/k/arrows/g/G/Home/End/`/`+filter); Placeholder/TooSmall â†’ drop. Placeholder tiles show a "Phase N â€” not yet implemented" hint when focused.
- Pty tile rendering: pty bytes from `Event::PaneOutput` / `Event::PaneSnapshot` feed a `vt100::Parser` owned by the App. The pty tile renderer projects the parser's screen buffer (cells + fg/bg colors + bold/italic/underline/reverse attributes) into ratatui cells within the tile's `Rect`. Cursor rendered as a reversed cell only when focused (unfocused tiles show the buffer without a misleading caret). Parser is resized on every Resize event to match the pty tile's new dimensions; corresponding `ResizePane` envelope sizes the daemon-side pane to match the tile (NOT the full terminal), so vim et al. render inside the box.
- Scope rendering contract (ARCHITECTURE Â§9): every scope renderer takes `(state, Frame, Rect, focused: bool)` and draws only within its `Rect`. Focused tile borders are bright cyan; unfocused are dim gray. `scope::docker::render` preserves all C2c2 rendering logic (three-state lifecycle: Connecting/Available/Unavailable, container table with `â–¶` selection marker, filter bar with `_` caret when active, help bar). `scope::placeholder::render` draws a bordered block with centered label + focused hint. Phases 4/5/7/9 inherit â€” their tile slots into the god view at its placeholder and is replaced by the live renderer when the phase lands.
- TUI action keybinds (C3a): focused Docker tile recognizes `r`/`R` (Restart) and `s`/`S` (Stop) for immediate dispatch, plus `K` (Kill) and `X` (Remove) which first enter an inline confirm modal per CTO Â§4. Dispatch allocates a request id via the shared monotonic allocator, inserts a `PendingAction { deadline: now + 30 s, description }` keyed by that id, and sends `Payload::DockerAction(DockerActionRequest { request_id, container_id, kind })`. Keybinds no-op when Docker is Unavailable or the visible list is empty; focus on a non-Docker tile routes the keys to the normal input path (PTY's `SendInput`, placeholder drop).
- TUI confirm modal (C3a, UX clarification #3): centered bordered box rendered *inside* the Docker tile's `Rect` â€” never full-screen, other tiles keep rendering and receiving live updates throughout. Prompt text: "`<Kill|Remove>` container `<name>`? [y] confirm Â· any other key cancels". `y`/`Y` clears the confirm and dispatches the `DockerAction`; any other key clears the confirm without dispatching; focus moving away from the Docker tile cancels; a 10 s idle deadline clears the confirm on the next Tick sweep. Help bar swaps to the confirm-specific hint while active.
- TUI toast overlay (C3a, UX clarification #2): rendered as a 1-line-per-toast strip positioned directly above the Claude Code tile (full scope-row width). `App::toasts: VecDeque<Toast>` is bounded to 3 â€” a 4th arrival `pop_front`s the oldest silently so the user's keystrokes are never gated on a toast. Per-kind auto-dismiss: Success ~3 s, Error ~8 s, Info ~4 s. Toasts are `Success`/`Error`-colored content with a short `ok:` / `err:` / `info:` prefix. The 30 Hz Tick drives the sweep that retires expired entries; state transitions that want to surface a toast call `App::push_toast(kind, message, &mut actions)` which bounds the queue, appends `AppAction::ShowToast { kind, message }` for the runtime's `tui.log` trace, and emits `DrawFrame`.
- TUI pending-action lifecycle (C3a): on every 30 Hz Tick, `App::sweep_expired(Instant::now(), â€¦)` walks `pending_actions` and drops entries whose `deadline` has passed, each expiring entry emits an "`<verb> <name>` timed out â€” check engine" Error toast. `DockerActionResult { request_id, kind, outcome }` looks up (and removes) the entry by `request_id`: `Success` â†’ green "`<verb> <name>` â€” succeeded" toast; `Failure { reason }` â†’ red "`<verb> <name>` failed: `<reason>`" toast (reason verbatim from dockerd); stale results (no matching pending action) fall back to `<verb> <container_id>` so the user still sees the outcome. `Payload::Error` from the daemon lands in the same overlay queue. `AppEvent::PendingActionTimeout(u64)` is retained on the event surface for a future dedicated timer sweeper without reshaping the API.
- TUI logs sub-state (C3b): `DockerScope.view` is a `DockerView::{List, Logs(LogsView)}` enum. `l` on the focused Docker tile allocates a sub id, sends `Subscribe(DockerLogs { id, container_id, follow: true, tail_lines: 0 })`, and transitions to `DockerView::Logs(LogsView { container_id, container_name, sub_id, lines: VecDeque, pending_stdout/_stderr: Vec<u8>, scroll_offset, at_tail, stream_ended })`. Events arriving on `LogsView.sub_id` are routed to `handle_logs_event` via `DockerScope::is_current_logs_sub`; `ContainerLog { stream, data }` appends bytes to the per-stream pending buffer and flushes every complete `\n`-terminated line (CRLF also stripped) into the capped ring (`MAX_LOG_LINES = 10_000`, drop-oldest). `DockerStreamEnded { reason }` flushes any trailing pending bytes as a final line, records the reason on `stream_ended`, and disables `at_tail`; the renderer paints a dimmed "â€” log stream ended: `<reason>` â€”" line at the tail. Scroll keybinds: `j`/`k`/Down/Up by one line; PgUp/PgDn by `LOGS_PAGE_LINES = 10`; `G`/End/Bottom jump to tail + re-enable auto-follow; `Esc` / `q` unsubscribe and return to List. Scrolling up sets `at_tail = false`; reaching offset 0 via scroll-down re-enables it. Logs view persists across focus moves; action keybinds (`r`/`s`/`K`/`X`/`l`/filter) are all ignored while logs are showing (read-only transcript). Stale events after Unsubscribe drop silently.
- TUI redraw cadence: 30 Hz `tokio::time::interval` always-on (no mode gating â€” the tile grid is always live). App responds to every `Tick` with `AppAction::DrawFrame`; the runtime draws the entire tile grid via ratatui's buffer diff (redundant-cell writes are cheap). State changes (pty output, container list update, focus move, filter edit) also emit `DrawFrame` directly so updates don't wait for the next tick.
- Daemon handshake: rejects protocol-version mismatches at the socket level. Both `Envelope.version` and `Hello.client_version` must equal the daemon's `PROTOCOL_VERSION`; otherwise the daemon sends a structured `Error(VersionMismatch)` naming both versions and closes. Without this guard a v3 client connecting to a v4 daemon would silently handshake and later trip an opaque rkyv decode error.
- Panes inherit `TEPEGOZ_PANE_ID=<id>` in env. TUI refuses to run if its own env has that var (prevents recursive attach feedback loop).
- Shells spawn in the TUI's `current_dir()` rather than `$HOME` (portable-pty's default).
- TUI sets the terminal window title to `tepegoz Â· pane N` on attach (OSC 0) and clears it on detach, so an attached pane is visually distinct from the outer shell.

## Phase 3 polish candidates (post-close follow-ups)
